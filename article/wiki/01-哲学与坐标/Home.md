# 01-哲学与坐标

这一组页面不讲具体礼法怎么操作，而先回答三个更根本的问题：为什么 AI coding 在深水区里会变成治理问题；为什么黑盒多-Agent 的风险不只是“代码可能写错”；以及 Cyber-Ming-Protocol 在公开世界的方法论坐标里究竟站在哪里。

如果说 `README.md` 负责立总纲，那么这一组页面负责把总纲背后的上位判断讲透。它的任务不是重复“帝王 Coding 很强”，而是把这套协议为什么成立、为什么不能被偷换成普通 workflow 建议，先在理论层和坐标层钉死。

## 这一组页面解决什么问题

- 为什么 AI coding 已经不再只是纯粹的 CS 问题
- 为什么黑盒多-Agent 在深水区会同时制造技术失真与治理失真
- 为什么这套协议不是凭空发明，也不是单纯的风格化包装
- 为什么它更适合被理解为压在 Codex / Claude Code 之上的治理层

## 推荐阅读顺序

如果你第一次进入这一组，建议按下面顺序读：

1. [为什么 AI Coding 已经模糊了 CS 与管理学的界限](为什么-AI-Coding-已经模糊了-CS-与管理学的界限.md)

3. [相关工作与方法论坐标](相关工作与方法论坐标.md)

这个顺序的逻辑是：

- 第一篇先解释为什么治理问题会被现实强行请进场
- 第二篇再解释黑盒多-Agent 为什么会在技术层和制度层同时塌陷
- 第三篇最后把 Cyber-Ming-Protocol 放回公开世界的方法论地图里，说明它与现有路线的共通与分叉

## 每一页各自承担什么功能

### 为什么 AI Coding 已经模糊了 CS 与管理学的界限

这一页讲的是位置变化。它解释的不是“管理学很重要”这种空话，而是：当执行位开始被部分外包给高吞吐数字执行体，开发者就不得不同时处理委派、路由、审计、接手与裁决这些治理问题。

### 黑盒多-Agent 的双重失真：技术失真与治理失真

这一页讲的是病灶结构。它把黑盒多-Agent 的风险拆成两层：技术失真让错误伪装成推进，治理失真让系统在错误出现时失去打断、追责、接手与重建秩序的能力。

### 相关工作与方法论坐标

这一页讲的是外部坐标。它回答“这套协议是不是只是换壳重述别人思路”这个问题，并说明它与 harness-driven、spec-driven、HITL、context engineering 共享什么，又在哪些制度层面真正分叉。
