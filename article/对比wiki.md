## 6.4 相关工作与方法论坐标

Cyber-Ming-Protocol 并非凭空出现。  
在公开世界里，已经有人分别从 spec-driven、harness-driven、human-in-the-loop、context engineering 等方向尝试约束 AI coding。  
本协议与这些工作共享若干基础认识，例如：上下文会腐烂、长时程任务需要工件与历史抓手、人类不应完全退出复杂软件开发。  
但它们的问题中心、制度设计与治理重心并不相同。

### 一、90% 以上相似：目前暂无公开可比对象

截至目前，尚未见到公开项目同时具备如下特征：

- 以深水区治理而非浅水区提效为核心问题
- 以“主权夺回”而非“生成增强”为主轴
- 明确区分执行位与审计位，并要求物理隔离
- 强调人类保留跨系统物理路由权
- 将 Git 起居注、Worktree 分封、双端续命、脉冲分封制整合为一套制度
- 将角色叙事上升为高摩擦协议的执行燃料

因此，从公开框架中，尚未发现与本协议高度同构的对象。

### 二、70%–90% 相似：长时程 Agent Harness 路线

最接近本协议核心问题的，是 Anthropic 关于 long-running agents harness 的公开工程思路。  
这一方向已经明确意识到：

- 长时程 Agent 会遗忘与失真
- 会试图一次做太多
- 需要 feature list、artifacts、progress file 与 Git history 来维持连续性
- 需要比“单个聊天窗口”更可靠的工程支架

这一点与 Cyber-Ming-Protocol 的“赛博起居注”“工件中枢”“轮转续命”确实高度相关。

但差异也很明显：

- 它仍主要在讨论如何让长时程 Agent 更可靠
- 而本协议更进一步，将问题提升为“如何在深水区保留人类主权”
- 它没有发展出明确的执行位 / 审计位双轨制度
- 也没有形成“角色叙事 = 执行燃料”的自我管理学层

因此，它与本协议在问题深度上高度接近，但在制度完整性与叙事层上仍有明显差别。

### 三、50%–70% 相似：Spec-Driven / Workflow-Driven 路线

这一层包括 GSD、OpenSpec、Spec Kit 等框架。  
它们的共同特点是：

- 强调规格、计划、上下文结构的重要性
- 反对让需求只活在聊天记录里
- 试图通过 spec / task / plan 让 AI coding 更有秩序

这与本协议的某些认识是一致的：  
AI coding 不能只靠“直接聊天 + 立即生成”。

但它们与本协议的根本差异在于：

- 它们主要解决“生成前如何把事情写清楚”
- 本协议主要解决“进入深水区之后如何不失控”
- 它们更像蓝图法
- 本协议更像朝廷法

换言之，Spec-Driven 路线主要在治理“生成前的清晰度”，  
而 Cyber-Ming-Protocol 主要在治理“生成中与生成后的主权、失真与重构抓手”。

### 四、知识概念相关：HITL、Context Engineering 与 Artifact 思维

再往下看，还有一些与本协议相关、但不足以构成框架级相似的方法与概念：

- Human-in-the-loop software development agents
- Context engineering
- Artifacts / Git history / progress file
- Approval / review gate

这些方向与本协议共享一些基础判断：

- 人类不能完全退出复杂开发
- 聊天历史不能充当唯一真相源
- 长链任务必须有工件和时序抓手

但它们通常还停留在“概念层”“论文层”或“局部工作流层”，  
尚未整合成一套以深水区主权治理为中心的完整制度。

### 五、我们的坐标

因此，Cyber-Ming-Protocol 既不是对现有 spec-driven 框架的简单改写，  
也不是对 harness / HITL 思想的平移复述。

它更接近一种新的方法论坐标：

- 以深水区治理为问题中心
- 以主权保留为制度核心
- 以双轨隔离、人类物理路由、Git 起居注、Worktree 分封、双端续命与脉冲分封制为方法骨架
- 以角色叙事与游戏化执行为高摩擦协议的持续动力

如果说主流框架主要在回答“怎么让 Agent 更能写”，  
那么本协议真正回答的是：

**当项目进入深水区之后，怎么让人类继续是皇帝，而不是逐步沦为给 Agent 善后的官吏。**
