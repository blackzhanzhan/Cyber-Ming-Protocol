# Cyber-Ming-Protocol Wiki写作手册

这份手册用于两个目的：

1. 固定当前已经达成共识的 Wiki 信息架构。
2. 在上下文腐烂或需要开新窗口接手时，作为新的写作中枢材料。

注意：这不是公开 Wiki 正文，而是本地写作总控文件。

---

## 0. 使用原则

- `README.md` 负责：总纲、入口、最小闭环、法统摘要、核心边界。
- Wiki 负责：哲学展开、方法论坐标、礼法细则、治理扩展、脱敏战报、样本页。
- Wiki 不重复 README 的总论，不把 README 改写成分页版。
- Wiki 每页只解决一个清晰问题。
- 公开页面一律业务去敏：代码可验证，证据可验证，业务不可直接识别。

---

## 1. 素材分级与可信度顺序

新窗口接手时，读取素材请按下面优先级判断：

### A级：当前总纲与已定稿表达

- `README.md`

用途：

- 统一术语口径
- 锁定项目对外自我定义
- 锁定最小闭环、五柱法统、边界与 FAQ 的现行版本

规则：

- 若 Wiki 文案与 `README.md` 冲突，以 `README.md` 为准
- Wiki 只展开，不推翻 README 已定稿表述

### B级：概念池与制度池

- `article/西苑代码治世秘录大纲.md`
- `article/辩经素材.md`
- `article/对比wiki.md`

用途：

- 抽取机制、哲学、对比坐标、边界判断
- 为 Wiki 页面提供可展开的概念骨架

规则：

- 大纲中的夸张定量、强修辞、未证实底层理论，不可直接写死进公开 Wiki
- 辩经素材中的判断，可提炼成更稳定的技术表达

### C级：证据池与战报池

- `article/示例.md`
- `article/开发效率示例.md`

用途：

- 提供战报、流程证据、伪完成识别样本、Git 起居注样本

规则：

- 绝不可直接原样公开
- 必须先去敏再写入战报层页面

### D级：发布与去敏策略池

- `article/plan.md`

用途：

- 只用于提炼：去敏策略、证据公开原则、发布时的可信度判断

规则：

- 不作为 Wiki 主体正文来源
- 不把其中的传播预期、平台策略写进 Wiki 主体

## 2. 本轮已确认、必须保留的新增决议

这些点不一定完整体现在仓库现有素材里，但已经在当前写作讨论中定下，后续新窗口必须保留：

1. `对比层不能消失`  
   它不单独裂成一级目录，但必须作为 `01-哲学与坐标/` 下的核心页保留。

2. `赛博探马机制是核心战术礼法`  
   它不是战报附属页，也不是调试技巧，而是“先试链路，再上大军”的前置治理法。

3. `业务去敏是公开层硬要求`  
   公开 Wiki 与战报页不得直接出现“考公 / 行测 / 资料分析 / 真实讲次 / 真实路径 / 真实平台 ID”等业务壳。

4. `认知债务不会被消灭，只能被治理`  
   AI 产码速度快于人类理解速度，认知债务放大是结构现实，不是个人失败。

5. `本协议作用不是填平认知债务，而是缩小认知债务剪刀差`  
   尤其包括：
   - 人类路由中的复制粘贴动作，在这里不是低级劳动，而是减小认知债务剪刀差的治理动作。
   - 当主观掌控力开始下降时，可以直接新开窗口，让执行位查阅赛博起居注回答问题，从而实现认知债务的灵活偿还、急速偿还、可信偿还。

6. `人的要求被改写了，但人不能高枕无忧`  
   这套协议不是让人退出战场，而是把人的负担从低维编码执行，改写为高维拆解、审计、路由、打断、验真、还债。

7. `不要把“人的新要求”写成整个 CS 行业的宏大议题`  
   页面必须聚焦：在本协议下，开发者需要什么能力结构，以及它与传统黑盒流派对人的要求有何不同。

## 8. 去敏硬规则（只在手册中固定，不单独立 Wiki 页面）

### 8.1 公开层绝不能直接出现

- `考公`
- `行测`
- `资料分析`
- 真实课程名 / 讲次名
- 真实平台 ID
- 真实本地路径
- 任何可直接指向原业务壳的标签

### 8.2 统一替换壳

优先替换为：

- `某垂直领域视频课程知识图谱`
- 或 `多模态时空知识图谱（视频字幕 + 图数据库 + 双链文档）`

### 8.3 去敏原则

- 代码可验证
- Git 可验证
- 错误链路可验证
- 业务不可直接识别

### 8.4 去敏不代表削弱证据

- 保留：流程链、错误链、提交链、验收链
- 抹去：真实业务壳、真实业务标签、真实路径与平台指纹

## 10. 新窗口接手协议

当上下文腐烂，或需要开新窗口继续写 Wiki 时，最小接手包如下：

### 必喂材料

1. `README.md`
2. `article/Wiki写作手册.md`
3. 当前要写的目标文件路径
4. 该页面对应的主素材文件

### 推荐喂法

新窗口起手时，直接说：

```text
你现在接手 Cyber-Ming-Protocol 的 Wiki 写作。

先读取：
1. README.md
2. article/Wiki写作手册.md
3. 本次目标页面对应的主素材文件

本次只写：<目标页面路径>

要求：
- 忠于 README 已定稿法统
- 只展开，不推翻
- 严格执行手册中的去敏规则
- 不得把这套协议降级成普通 AI 协作最佳实践
- 如果素材不足，明确说“这里素材不足，建议暂不写死”
```

### 不要只喂单页素材

如果只给新窗口一篇素材而不喂 `README.md` + 本手册，新窗口很容易：

- 把法统写散
- 把边界写虚
- 把帝王叙事削平
- 把战报写成普通案例复盘

## 11. 最后提醒

这套 Wiki 的目标不是“把 README 变长”，而是：

- 把总纲后的机制拆开讲清
- 把最值钱的独特机制固定下来
- 把脱敏后的证据链写成可验证兵书
- 让新窗口在上下文腐烂后，仍能沿着同一法统继续写下去

如果后续页面新增，先问自己三件事：

1. 这是总纲该说的，还是 Wiki 该接手的？
2. 这是机制页、哲学页、扩展页，还是战报页？
3. 这页有没有足够素材支撑，还是暂时只够写成边界？

守住这三问，Wiki 就不会散。
