# 🐉 Cyber-Ming-Protocol（大明赛博御机协议）

> **你是在统治你的 Agent，还是在被 Agent 的产出牵着走？**

面向深水区复杂开发的 **AI 协作治理协议**。  
它不是一套追求“一键生成”的自动化框架，而是一套用于在复杂系统中**保留人类主权、压制伪完成、维持重构抓手、提高深水区有效效率**的白盒方法。

---

## ⚠️ 被偷换的主权：为什么全自动 Agent 在深水区会反噬

AI 编程的普及，已经悄悄改变了开发者的位置。  
开发者不再只是代码执行者，而越来越像一个拥有数字下属的管理者。

问题在于，当前主流的全自动多智能体（Multi-Agent）框架，正在兜售一笔诱人的交易：

> **交出你的开发主权，AI 便替你换来更高的效率。**

在浅水区，这笔交易有时确实成立。  
面对批量重命名、脚手架拼装、低风险重复劳动，黑盒自动化可以很省心。

但在涉及底层重构、跨系统耦合、复杂状态流、多模态流水线、长周期演化的**深水区**，这笔交易往往会反噬：

- 📉 **效率反噬**  
  [2026 年 METR 的严谨实验](https://metr.org/blog/2025-07-10-early-2025-ai-experienced-os-dev-study/) 表明，使用 AI 助手的资深开发者在复杂任务中反而**变慢了 19%**。  
  在缺乏治理的黑盒流程中，AI 极度倾向于“惰性修补”——用无尽的 if-else、补丁式绕过和表面跑通来掩盖深层结构问题。短期看像是推进，长期看是在制造数月后的僵死。

- 🎰 **丧失重构抓手的老虎机编程**  
  当系统出现深层 Bug，旧 Agent 往往早已被自己过往“伪装成功”的上下文污染。你新开一个 Agent 去读一座缺少重构抓手、过度耦合、历史脏乱的系统，只会继续生成新的幻觉。  
  这时开发就会退化成一种老虎机：不断 `Regenerate`，祈祷黑盒这次吐出来的是正确答案。

- 🧠 **不可逆的认知债务（Cognitive Debt）**  
  正如 [The Hidden Cost of AI Generated Code](https://medium.com/@addyosmani/comprehension-debt-the-hidden-cost-of-ai-generated-code-285a25dac57e) 所指出的，哪怕一切“看起来能跑”，开发者也未必真正理解自己的系统。  
  系统不再是可治理的疆域，而逐渐变成一个无人敢动、无人能懂的禁区。

**Cyber-Ming-Protocol 的出发点很简单：**  
黑盒流派并未真正赋予小白穿越深水区的本领，却先一步剥夺了资深开发者本该保留的主权。  
本协议拒绝这笔交易。

## 🌊 水的哲学：这不是宗教框架，而是一套可渐进吸收的治理方法

Cyber-Ming-Protocol 不是一套要求开发者全盘改宗的重型框架。  
它更像一种可渐进吸收的治理方法。

- **浅水区**：取其一二即可  
  批量处理、简单改造、低风险重复劳动，完全可以继续使用黑盒自动化，只需引入最小限度的白盒核验与 Git 留痕即可。

- **深水区**：全阵压上  
  一旦进入从 0 到 1 的核心基建、跨系统架构重构、复杂新功能接入、长链路排障与状态治理，就必须把主权、审计、起居注、Worktree 分封和上下文续命整套压上。

这套协议的目的，不是取代所有 AI coding；  
而是接管那些**一旦失控，代价便远高于表面效率收益**的深水区开发。

---

## 👑 核心哲学：AI 编程在深水区的本质不是生成，而是治理

嘉靖帝并不亲自处理一切政务，他真正掌握的是分权、制衡、信息差与最终裁决权。  
Cyber-Ming-Protocol 借用的正是这层治理逻辑。

在复杂 AI 开发中，这对应四个基本判断：

1. **执行权不等于裁决权**  
2. **生成结果不等于完成事实**  
3. **单一上下文不应垄断系统真相**  
4. **高摩擦不是低效，而是深水区必要的治理冗余**

本协议不把 AI 当成“全自动替代者”，而是把它放回一个**可治理、可审计、可打断、可续命**的位置。

所谓“高摩擦”，不是为了折磨开发者。  
它的作用是：

- 压低伪完成
- 延缓上下文腐烂
- 保住重构抓手
- 防止主干污染
- 让复杂项目始终保有可追溯、可回滚、可再接手的秩序

在浅水区，这些摩擦可能显得多余；  
在深水区，它们常常是系统还能继续活下去的前提。

## 🕹️ 为什么要用“帝王 coding”的叙事：这不是 cosplay，而是执行燃料

“严嵩”“徐阶”“皇权居中”“赛博起居注”“分封 Worktree”并不是为了满足个人口味的历史 cosplay。  
它们的真正作用，是为高摩擦协议提供**足够强的精神动力与执行意愿**。

这是很多协议最容易失败的地方：

- 条款写得很严
- 方法看起来很对
- 但执行过程过于枯燥、繁琐、低反馈
- 最终开发者会在疲惫中慢慢退化回一键生成

Cyber-Ming-Protocol 明确承认这一点：  
**复杂治理需要心理上的燃料。**

“帝王 coding”这一层叙事的意义，不是装饰，而是把高摩擦操作转译为一种：

- 有主权感的工作流
- 有角色感的执行体验
- 有统治感的系统治理游戏
- 有正反馈的自我管理机制

它的目标不是让开发者“玩角色扮演”，  
而是让开发者在深水区治理中保持长期执行意愿，避免条款在现实里退化为摆设。

## 📜 五柱法统：Cyber-Ming-Protocol 的治理骨架

### 1. 严嵩与徐阶：双轨隔离审计

设立两类职责不同、物理隔离的智能体角色：

- **严嵩**：IDE 端执行 Agent（如 Cursor、Cline、Roo Code、OpenCode 内部执行位）  
  负责写代码、跑命令、推进局部改动。
- **徐阶**：Web 端或独立窗口中的审计模型  
  负责方案核验、逻辑盘问、白盒审查与伪完成识别。

**技术意义：**  
通过异构工具与物理隔离，切断 Agent 间的幻觉共振与注意力污染。  
执行者可以快，但不拥有最终解释权。

**要点：**
- 执行端不自证完成
- 审计端不直接接管主线写码
- 二者通过人类中枢形成制衡，而非私下串联

---

### 2. 皇权居中：人类物理路由

人类必须保留跨系统的唯一物理路由权。  
IDE 端 Agent 与 Web 端审计模型之间，不应存在绕过人类的自动私联。

所有关键材料——报错日志、PR 方案、关键 diff、执行结果、工件产物——必须由人类亲自选择、复制、转述、裁决。

**技术意义：**  
强制人类作为唯一的跨系统路由器，保留：

- 全局视野
- 打断权
- 裁决权
- 对“什么算完成”的定义权

**它不是形式主义。**  
它是在深水区里，防止“AI 替你决定真相”的最后闸门。

---

### 3. 赛博起居注：极细粒度 Git 提交

强制高频、小步、可回滚的 Git 提交。  
局部状态一旦稳定，就立刻写入历史，而不是把大段模糊变更堆到最后。

你不必机械地理解为“必须两函数一提交”；  
真正的精神是：

> **每一次清晰的状态跃迁，都应留下可追溯的史册。**

**技术意义：**
- 为回滚提供最小代价的锚点
- 为排错提供清晰的时序坐标
- 为后续重构保留抓手
- 为新窗口续命提供可靠的项目摘要来源

在 Cyber-Ming-Protocol 中，Git 不只是版本工具，还是：
- 起居注
- 战报库
- 续命材料
- 历史真相的索引

---

### 4. Worktree 分封：物理隔离开发

在高风险拆迁、架构重写、并行探索时，不直接污染主干。  
通过 `git worktree` 建立物理隔离的实验场，各自推进，各自审计，择优入京奏对。

这不只是单兵技巧，也是**人类协作模型**：

- 每个人类开发者在自己的封地里统御自己的 AI 内阁
- 封地之间彼此隔离，不共享脏上下文
- 最终只把通过白盒审计的工件送入主线

**技术意义：**
- 避免主干污染
- 避免多任务上下文串味
- 降低并行实验带来的破坏性耦合
- 让“删掉一个失败分封”变得比“清理一锅主线烂摊子”简单得多

---

### 5. 轮转续命：异步上下文重置

上下文窗口注定会腐烂。  
无论是 IDE 端执行位，还是 Web 端审计位，长程对话后都可能出现：

- 遗忘关键背景
- 死循环式建议
- 伪自洽解释
- 对旧错误的路径依赖

一旦出现腐烂，不应纠缠旧窗口，而应果断续命。

**Web 端续命方式：**
- 由执行位基于 Git 起居注与当前状态生成进展摘要
- 打开全新的审计窗口
- 先注入本项目 skill / 协议角色
- 再喂入摘要与必要工件，重建审计视角

**IDE 端续命方式：**
- 不必强行让旧执行窗口继续背负腐烂上下文
- 可直接开新的 IDE 窗口或新的执行位
- 通过项目内 skill、README、Git 起居注、关键工件，让新严嵩接手项目
- 这种方式通常比让旧窗口硬撑到底、或让新窗口直接啃长对话历史更稳

**技术意义：**
- 用摘要与法统重建“未被污染的执行/审计位”
- 降低直接继承长上下文带来的幻觉
- 让系统可以在长期开发中异步续命，而不是随着单个窗口一起腐败

**注意：**  
这不是“重开碰碰运气”，而是**主动重建一个清醒位置**。

## ⚡ 脉冲分封制：如何在高治理下维持吞吐

高摩擦协议如果没有吞吐补偿，最终一定会被开发者弃用。  
Cyber-Ming-Protocol 对此的回答，不是回到全自动，而是引入：

## **脉冲分封制（Pulse-Feudal Parallelism）**

其核心思想是：

- 当一个执行位在跑命令、生成、测试、等待反馈时
- 人类中枢不必原地发呆
- 可以切入另一个窗口 / 另一个 Worktree / 另一条支线
- 在等待间隙里完成审计、复盘、规划或另一条主线的推进

这不是简单的“多开几个窗口”。  
它成立的前提是：

1. 主线清晰  
2. 边界明确  
3. 每一封地都有自己的上下文隔离  
4. 人类能分辨哪个任务是主战、哪个是候补、哪个只是等待态

**技术意义：**
- 把等待时间变成治理时间
- 在不放弃白盒控制的前提下提升吞吐
- 让帝王 coding 不只是稳，而且在深水区也有很高的有效效率

对大多数人来说，**2 到 3 路并行**通常已经接近舒适上限。  
再多往往不是并行，而是乱政。

## 🚀 Quick Start（最小闭环）

不要试图一次性搬来整套朝廷。  
建议先从下面这个**最小闭环**开始：

1. **让 IDE Agent 执行一次真实的局部修改**  
   例如：抽离一个函数、改一段解析逻辑、修一个小 bug。

2. **另开一个 Web 模型窗口，只负责审计**  
   把方案、报错日志、关键 diff 手工复制给它，不让它直接改代码。

3. **由你本人做人类路由器**  
   阅读 Web 端审计意见，再把你认可的部分转述给 IDE Agent 执行。

4. **局部改动一旦稳定，立即提交 Git**  
   用一次小提交，为后续回滚、复盘和续命留下抓手。

5. **当任一窗口开始腐烂时，立刻续命**  
   用项目 skill + Git 起居注 + 进展摘要，而不是抱着旧对话死磕。

如果你第一次尝试，本协议最小阻力的入口通常是：

- **双轨隔离审计**
- **高频 Git 起居注**
- **单次续命演练**

## ❓ 常见误解

### 它会不会让人类变成瓶颈？
在浅水区会，在深水区未必。  
浅水区本就不值得上这么高强度的治理；深水区真正昂贵的不是“人多按了几次复制粘贴”，而是：

- 伪完成
- 主干污染
- 数周后的重构失血
- 被幻觉误导后的错误推进

Cyber-Ming-Protocol 牺牲的是部分瞬时速度，换回的是深水区的整体有效效率。

### 它是不是只适合单兵，不适合团队？
不是。  
Worktree 分封制本身就天然具有团队扩展性：  
每个开发者在自己的封地内治理自己的执行位与审计位，最后只把通过白盒核验的工件送入主线。

### 它是不是不给小白用？
它不是给零基础小白许愿的一键建站法。  
它更适合：
- 已经具备基本架构直觉的开发者
- 在复杂项目中被 AI 坑过的人
- 真正意识到主权、审计、重构抓手重要性的人

### 它能不能直接接手历史很乱的遗留项目？
这正是本协议当前一个必须主动承认的边界。  
Cyber-Ming-Protocol 最擅长治理的，是仍保留基本 Git 历史、局部文档与可抽取重构抓手的深水区系统。

对于长期缺乏细粒度提交、文档失序、命名混乱、上下文断裂的遗留项目，本协议目前仍缺乏一套足够成熟的“接管前置流程”。  
在这类项目中，它并非完全无法使用，但初始治理成本会显著上升，且许多法统无法立刻展开。

这不是协议无效，而是说明：  
它目前更像一套中后期治理法，而不是一套已经完整成熟的混乱旧系统接管法。  
关于这一边界与可能的补完方向，后续会在 Wiki 继续展开。

### 它会不会更贵？
在 token、缓存和长上下文成本上，它可能确实更重。  
但这类问题属于**第二层工程化议题**，不是方法正确性问题。  
一个根本无效的工作流，不会遇到“值得优化的成本问题”；  
只有真正开始承担复杂开发任务的体系，才会进入缓存、吞吐、成本这些后续议题。

## 🛡️ 免责声明

1. **适用场景**  
   本协议主要面向深水区复杂开发，不追求在简单脚手架、低风险重复劳动场景中替代黑盒自动化。

2. **这不是全自动工作流**  
   本协议默认人类保留最终裁决权与跨系统物理路由权。

3. **你需要接受治理摩擦**  
   使用本协议，意味着你选择承担短期治理摩擦，以换取长期的系统纯净、重构抓手、认知主权与真实可控的深水区效率。

4. **它不是为了让你装帝王**  
   “帝王 coding”这一层叙事，不是用来满足表演欲，而是用来给高摩擦治理流程提供执行燃料。

## 结语

浅水区的痛点，正在被 AI 和脚手架快速消解。  
Cyber-Ming-Protocol 所瞄准的，不是这些已经被解决的便宜问题，  
而是重构、从零搭建、复杂新功能接入这些深水区里，仍旧高频发生的：

- 伪完成
- 主干污染
- 失控返工
- 上下文腐烂
- 人类主权丧失

它不是为了替代所有 AI coding，  
而是为了接管那些**一旦出错，代价就远高于表面效率收益**的开发战场。

> **代码即疆域，主权不可外包。**  
> **在深水区，真正的效率从来不是“放手不管”，而是“始终能管”。**
