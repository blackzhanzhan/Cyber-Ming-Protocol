# 安装与最小使用

## 这一页解决什么问题

这页只讲两件事：

- 当前仓库里哪些是可接入的 IDE-side skills
- 如果你要开始用，最小接入顺序应该是什么

它不负责承诺一个 repo 里并不存在的统一 installer。

## 先把浅尝试与深接入分开

- 浅尝试：只给仓库链接与授位文件，不要求安装
- 深接入：若宿主支持 skill 目录，默认只发生在 IDE 执行位
- Web 审计位默认按协议与模板工作，不以安装执行型 skill 为前提

## 当前仓库里有什么

### `skill/`

这里放的是 repo-side IDE skills：

- `global_rules`
- `approval-first-planner`
- `approved-checklist-executor`
- `probe-first-scout`
- `legacy-project-handover`

### `web-audit-templates/`

这里放的是 Web 侧审计模板：

- `plan_audit_template.md`
- `completion_audit_template.md`
- `succession_judge_template.md`

这三个模板不是本地 Skill，不属于 IDE-side 安装流程。

## 为什么这里不写“一键安装”

因为当前不同 IDE / agent host 对 skill 的装载方式并不统一。

也就是说，这个仓库当前提供的是：

- 可公开分发的 skill definitions
- 可公开协作的 Web 审计模板

但不是一个绑定单一宿主环境的统一安装脚本。

## 最小接入顺序

如果你已经决定进入深接入，再按下面顺序来，不必一上来全装。

### 第一步：先接入主干三件套

- `global_rules`
- `approval-first-planner`
- `approved-checklist-executor`

这三者解决的是最核心的问题：

- 先规划，再执行
- 规划必须可审批
- 执行必须按片、按验收、按提交归档

### 第二步：再按需要补扩展件

- `probe-first-scout`：当你面对的是真不确定，不是假复杂时
- `legacy-project-handover`：当你要新窗接手、上下文续命或只读交接时

## 最小使用顺序

即使接入 Skill，最稳妥的最小顺序仍然是：

1. 先让 planner 产出原子清单与边界
2. 让人类审核并决定是否准奏
3. 准奏后再让 executor 按片执行、验证、归档
4. 需要时用 Web 模板辅助方案审计、完成审计、续命判断

## 如果你的环境不支持 Skill

那就先不要被安装问题卡住。

你仍然可以完整实践协议：

- 手工要求 planner 风格的输出
- 手工要求 executor 风格的执行与证据回报
- 手工用 Web 模板组织审计位提问

Cyber-Ming 的立场一直是：**协议优先于安装。**

## 当前最容易踩的坑

### 坑一：一装 Skill 就想跳过理解

这样通常只会把误用变得更稳定。

### 坑二：一装 Skill 就想跳过审批

不对。
Skill 强化的是动作骨架，不是用来绕开人类批准与证据门槛。

### 坑三：把 Web 模板也塞进本地 Skill 流程

不对。
Web 模板属于 Web 审计位的协作资产，不属于 IDE 执行位的本地安装物。
