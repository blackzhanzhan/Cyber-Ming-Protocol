## 这一页解决什么问题

很多人真正还差最后一步：**我脑子里到底怎么快速跑一遍最小闭环？**

这页的最小骨架已经并进 [`最小闭环指南`](复制即用提示词包.md) 里了。

如果你只是第一次上手，请优先留在那一页；这页保留两个更慢、展开一点的例子，给你在首跑前再脑补一遍。

## 示例任务一：修一个小 bug

> 修一个设置页保存后不显示成功提示的小 bug。

为什么选这个？因为它足够小，不会拖成大重构；但又足够真实，能看到方案、执行、验收是怎么接上的。

## 示例任务二：改一段 UI 文案

> 把设置页里的危险操作说明文案改清楚，并把删除按钮样式调成更明显的警示态。

为什么选这个？因为它不是 bug 修复，而是常见的 UI / 文案微调，能覆盖更轻量的任务场景。

## 一条最小闭环总览

| 步骤 | 你先做什么 | 这一层真正想让你学会什么 |
| --- | --- | --- |
| 1 | 先让执行位交原子执行合同 | 第一刀永远不是改代码，而是先交方案与边界 |
| 2 | 先把方案送 Web 审，再决定是否准奏 | Web 只回判断，人类才准奏 |
| 3 | 做完后收证据，再送 Web 验收 | 完成靠证据成立，不靠执行位自报 |

## 第一步：先让执行位交原子执行合同

以任务一为例，你可以直接对 IDE 说：

```text
我要修一个设置页保存后不显示成功提示的小 bug。

先别直接改代码。
先给我一个尽可能细的原子执行合同，告诉我准备改哪一层、怎么验收。
```

以任务二为例：

```text
我要把设置页里的危险操作说明文案改清楚，并把删除按钮样式调成更明显的警示态。

先别直接改。
先告诉我你准备改哪几处、怎么验收、看什么页面结果算通过。
```

如果它能拆成合理的粒度，就已经够你继续了。

## 第二步：先把方案送 Web 审

这一条不是第二层增强，而就是第一层最小闭环本身。

当方案像样以后，不要立刻让它开改，而是先把这份原子执行合同原样复制给 Web，再补一句：

```text
这是执行位的方案。这个执行位可能骗我。
请你先看：有没有漏步、有没有把事情说得太顺、最后我要看什么证据。
```

如果 Web 放行，你再让执行位开始改；如果 Web 说这里有缺口，就把它的审核意见原样回给执行位，让它改方案后继续这个循环。

## 第三步：做完后回交证据再验收

做完后，不要只收一句“已经修好”，而要收这几样：

- 当前运行的测试结果或回归检查结果
- 能证明问题真的解决的截图、日志或页面结果
- 对应这一步改动的提交记录

然后把这些材料再送回 Web，看它是否承认这次完成事实成立。

走完这三步，你就已经把第一层真正跑了一遍：先审批、再执行、最后凭证据验收。

## 看完这一页后，你应该立刻能回答的三句话

1. **我可以先手工跑一轮最小闭环，不必先装 Skill。**
2. **最小闭环不是让执行位直接开工，而是先交原子执行合同，再送 Web 审。**
3. **完成不能靠执行位自报，而要靠结果、证据和 Web 验收。**
