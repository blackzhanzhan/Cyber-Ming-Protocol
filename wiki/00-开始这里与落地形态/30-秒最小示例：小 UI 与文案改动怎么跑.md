## 这一页解决什么问题

前一页的 30 秒示例展示的是一个小 bug 怎么跑。

但很多人第一次真正会拿来试的任务，其实不是修 bug，而是更轻的小改动：

- 改一段提示文案
- 调一个按钮样式
- 让一个页面的危险态更清楚

这一页就是专门用这种任务面，演示 Cyber-Ming 怎么在 30 秒里被脑补完一轮。

## 示例任务

这次固定用一个很小的任务：

> 把设置页里的危险操作说明文案改清楚，并把删除按钮样式调成更明显的警示态。

为什么选这个？因为它不是 bug 修复，而是非常常见的 UI / 文案微调；它足够小，不会拖成整页改版，但又能看出边界、验收、证据和复核是怎么接上的。

## 最适合展示的证据载体

这类任务最适合展示的，不是长日志，而是当前页面的可见结果。

最稳的证据通常是：

- 当前页面的前后对比截图
- 如果涉及交互态，再补一段很短的录屏或状态截图
- 对应这一步改动的 diff 或提交记录

如果你的环境里刚好有视觉回归、组件快照或页面测试，那当然更好；但对这种任务来说，最硬的第一层证据仍然是**当前页面到底长成什么样**。

## 三种跑法总览

| 跑法 | 你先做什么 | 它最能说明什么 | 不是在说明什么 |
| --- | --- | --- | --- |
| 不装 Skill | 先让 IDE 交清单与验收点 | 协议也能覆盖轻量 UI / 文案改动 | 只有大工程才值得用这套方法 |
| 装了 Skill | 用 skill 稳定规划与执行节奏 | Skill 让边界更稳 | Skill 会替你判断页面是不是更好 |
| Web 审计固定在循环里 | 把方案先交给 Web 审，再决定是否执行 | 审计流程不丢失，同时减少人类反复组织流程的精力损耗 | Web 位会变成第二个执行位 |

## 第一种：不装 Skill，手工跑

### 第一步：先让执行位交清单，不准直接开改

你可以直接对 IDE 说：

```text
我要把设置页里的危险操作说明文案改清楚，并把删除按钮样式调成更明显的警示态。

先别直接改。
先告诉我你准备改哪几处、怎么验收、看什么页面结果算通过。
```

如果它能把任务拆成类似下面这种粒度，就已经够你继续：

- 定位当前危险说明文案在哪一块，准备改成什么
- 调整删除按钮的颜色、边框或强调态，让危险性更明显
- 打开设置页核对文案和按钮样式是否都按预期显示

### 第二步：先把清单送 Web 审，再决定是否执行

这类任务虽然轻，但流程仍然一样：先把执行位的清单交给 Web，并明确告诉它这里可能有诈。

如果 Web 放行，执行位再开始施工；如果 Web 不放行，就把 Web 的审核意见原样回给执行位，让它改方案后继续下一轮。

### 第三步：施工结束后，再把页面证据送回 Web 验收

这类任务做完后，不要先听“已经更清楚了”，而要先收：

- 当前页面截图
- 如果有必要，再看一段很短的状态录屏
- 对应这次改动的提交记录

这时你真正验的，不是“代码是不是改过”，而是：

- 危险说明是不是更清楚
- 删除按钮是不是更显眼
- 页面有没有被顺手改乱别的东西

## 第二种：装了 Skill，边界更稳但不替代页面证据

同一个任务放到已经接入 Skill 的环境里，变化主要体现在边界和节奏更稳定。

### planner 更容易先把边界压清

像 `approval-first-planner` 这类 skill，更容易先把任务压成：

- 只改哪一段文案
- 只改哪一个危险按钮样式
- 用什么截图或页面结果做验收

这样会减少“顺手把半个设置页一起重写了”的漂移。

### executor 更容易按片验证和归档

像 `approved-checklist-executor` 这类 skill，更容易把后续动作守成：

- 先按批准范围施工
- 先看页面结果
- 再提交归档

### 但最后还是要看页面有没有真的变对

对这类任务来说，Skill 再稳，也不能替代页面证据。

你还是要看：

- 当前截图或录屏
- 对应提交记录
- 页面是否只在该变的地方变了

也就是说，Skill 让过程更稳，但不替代你对页面结果的判断。

## 第三种：Web 审计位固定在循环里

这一条也不是按需才开的增强项，而是这套协议哪怕在轻量页面任务里也不丢审计流程的关键。

### Web 位先看什么

Web 先看执行位交上来的方案，并明确知道：这个执行位可能把页面变化说得比实际更好看、更安全、更克制。

它先查的是：

- 文案改动有没有说清边界
- 危险态是不是只改该改的地方
- 最后该拿什么页面证据来验收

### Web 没通过时怎么走

如果 Web 说边界漂了、文案还不够清、危险态表达不够准，就把这份审核意见原样回给执行位，让它改方案，再继续下一轮方案审。

这样高摩擦并不意味着人类每轮都要重做一遍判断框架。人类真正做的是：把方案送去审、把审核意见回灌、把结果再送去验。

### Web 通过后还要看什么

Web 放行以后，执行位才按清单施工、验证、提交。

执行完成后，你再把这些材料送回 Web：

- 改前后截图或短录屏
- 这次文案与样式的变更点说明
- 对应提交记录
- 你自己最怀疑的风险点

Web 位此时更适合回答的是：

- 文案是不是确实更清楚
- 危险态是不是更明显
- 有没有顺手把不该动的页面部分也带偏

### Web 位不做什么

它不该：

- 顺手替你继续设计整个页面
- 把这个小改动扩写成大改版方案
- 不看截图和提交，只听总结就放行

它的职责仍然是独立复核，而不是第二个执行位。

## 看完这一页后，你应该立刻能回答的三句话

1. **不装 Skill，也能跑这类小 UI / 文案改动。**
2. **装了 Skill，会让边界和节奏更稳，但不会替代页面证据。**
3. **无论任务多小，方案都先送 Web 审；不过就回灌执行位改方案，通过后再执行提交，结果与页面证据还要再回 Web 验收。**
