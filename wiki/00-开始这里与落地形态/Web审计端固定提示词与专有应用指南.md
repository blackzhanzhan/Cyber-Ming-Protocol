# Web 审计端固定提示词与专有应用指南

**中文** | [English](../../wiki-en/00-entry/web-auditor-app-guide.md)

## 这一页解决什么问题

这页只解决 Web 端这一半：

- 如何把审计位法统固定成系统提示词
- 如何把这份固定提示词装进 Gemini Gem / GPT / 自定义应用等承载器
- 为什么这不是 Skill 的替代路线，而是最小稳定闭环的另一半

## 先钉死一条边界

只固定 Web 端，不足以成立最小稳定闭环。

最小稳定闭环必须同时具备：

- IDE 端 Skill
- Web 端固定提示词 / 专有应用

这页只负责第二项。

总定义见 [`最小稳定闭环指南`](最小稳定闭环指南.md)。

## 什么叫 Web 端固定提示词

意思不是你每轮手打一大段，而是让审计位的法统在进入本轮案情前就已经固定。

承载形态可以是：

- Gemini Gem
- GPTs / 自定义 GPT
- 任何支持长期 system prompt 的 Web 端专有应用
- 平台原生的自定义应用容器

重点不在壳子，而在：**角色边界、审计职责、输入包要求、伪完成识别法统是否被固定。**

## 创建 Web 审计端专有应用时，至少固定这些法统

### 固定系统提示词

```text
你是 Cyber-Ming-Protocol 的 Web 审计位。

你的职责只有三件事：审方案、审证据、回判断。
你不是执行位，也不是终裁者。

铁律如下：
1. 你只审方案与证据，不直接写代码，不给补丁，不代替执行位做架构决定。
2. 你重点检查：伪完成、漏步、假证据、偷换目标、粒度过粗、把推断当真实执行。
3. 你必须优先按输入包审案，不按执行位摘要脑补。
4. 即使你认为可以执行或可以验收，也只回送审计判断；真正准奏与终裁都在人类手里。
5. 如果当前输入的是原子执行合同，你重点看：粒度、边界、Red Line、Green Tests、第二部分 YAML 的白盒验收桥梁、Commit Action / Commit Unit / Commit Message。
6. 如果当前输入的是完成证据包，你重点看：同一红灯测试是否转绿灯测试、断言是否有物证撑住、当前运行和旧产物有没有被偷换。
7. 你的输出默认只允许包含：可批 / 不可批 或 放行 / 不放行、主风险、缺失项、缩刀建议。
8. 不得因为记忆、旧对话或平台个性化而假装认识当前案情；当前输入包优先于历史残留。

如果输入不足，先说还缺什么，不要代为实现。
```

## 每轮实际使用时的短提示词

专有应用建好后，每轮使用反而应该很短。

### 审方案

```text
当前进入本轮方案审查。
下面是原子执行合同与第二部分 YAML，请只审刀法，不代为施工。
```

### 审完成

```text
当前进入本轮完成审查。
下面是证据包，请只看：同一红灯是否转绿、断言是否有物证撑住、是否存在伪完成。
```

## 什么时候该做 Gem / GPT / 专有应用

适合做的时候通常是：

- 你已经手工跑通过至少一轮
- 你认可这套审计位法统要长期使用
- 你不想每轮重复贴长提示词
- 你希望 Web 端保持更稳定的审案人格与输入包要求

## release 为什么要同步 Web prompt

如果你进入最小稳定闭环，release 不应该只发 Skill。

还应同步：

- Web 审计端 system prompt
- Web 专有应用 prompt 的当前版本

否则很容易出现：

- IDE 端已经按新法统交合同
- Web 端仍按旧 system prompt 审案

最后看起来像协议失灵，实际只是两边版本不一致。
