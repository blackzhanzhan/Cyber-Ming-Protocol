## 这一页解决什么问题

这页里最短的 Skill 接入路径，已经并进 [`最小稳定闭环指南`](最小稳定闭环指南.md) 了。

如果你只是要把第二层搭起来，请优先留在那一页；这页只在你想把 IDE 端那一半单独拆开慢看时再来。

它只解决两个问题，但只限 IDE 端：

- **何时装**：你现在该不该装 Skill
- **如何装**：如果要装，按什么顺序来

核心原则：**先跑通协议，再决定如何把 IDE 端稳定下来。**

这页不是“最小稳定闭环”的总指南。

最小稳定闭环要求两边同时就位：

- IDE 端 Skill
- Web 端固定提示词 / 专有应用

总定义先看 [`最小稳定闭环指南`](最小稳定闭环指南.md)。

## 先给结论：何时装

### 适合先不装 Skill 的情况

- 你还在理解协议，尚未确认会长期采用
- 你只是想先读理念、战报与边界
- 你想先用自己现有的 IDE / Web 组合手工跑一遍
- 当前任务很小、很轻、很短，不值得引入完整治理骨架

### 适合开始装 Skill 的情况

- 你已经理解审批先行、原子执行合同、证据闭环三件事
- 你发现自己经常漏步骤、漏边界、漏证据
- 你想稳定 planner / executor / probe / succession 的动作
- 你经常在 AI coding 里遇到伪完成、惰性修补、上下文腐烂

### 暂时不建议全装的情况

- 你的宿主环境对 skill 触发支持还不稳定
- 你还没想好 IDE 执行位与 Web 审计位怎么分工
- 你目前只需要最小闭环，不需要完整的技能面板

### 一个简单判断法

如果你最近的主要痛点是：

- **不确定值不值得学**：先别装
- **知道该怎么做，但总忘记说全**：可以开始装
- **环境还很乱，连角色分工都没定**：先别全装
- **已经在复杂项目里反复踩伪完成和腐烂上下文**：尽快装核心 Skill

## 当前仓库里有什么

### `skill/`

这里放的是 repo-side IDE skills：

- `global_rules`：全局规则，定义不可违反的红线
- `approval-first-planner`：先交方案，不准直接开工
- `approved-checklist-executor`：按批准的清单执行、验证、归档
- `probe-first-scout`：不确定时先探测，不装懂
- `legacy-project-handover`：新窗接手、上下文续命、只读快照

### `web-audit-templates/`

这里放的是 Web 侧审计模板：

- `plan_audit_template.md`：审方案
- `completion_audit_template.md`：审完成
- `succession_judge_template.md`：判续命

这三个模板**不是本地 Skill**，不属于 IDE-side 安装流程。

## 最小接入顺序：如何装

### 浅尝试 vs 深接入

- **浅尝试**：只给仓库链接与授位文件，不要求安装，不要求 clone
- **深接入**：若宿主支持本地仓库操作与项目级 skill 目录，才进入 IDE 端接入步骤

浅尝试默认不 clone。先把 GitHub 仓库链接当远程法统来源来读；只有在你已经决定进入最小稳定闭环，才让执行端 agent clone 仓库并接入项目级 Skill。

### 第零步：先 clone 仓库，再谈 Skill

如果你要进入最小稳定闭环，执行端 agent 至少要能做到：

- 本地已有当前仓库，或能自己 clone 当前仓库
- 能读取 `skill/` 目录中的核心三件套
- 能按宿主机制把这三件套接入当前项目

最小动作就是：**先 clone 仓库，再接 skill。**

### 给执行端 agent 的接入提示词

```text
你现在进入 IDE 端稳定接入阶段。

如果你的宿主支持本地仓库操作与项目级 skill：
1. 若本地尚无仓库，先 git clone https://github.com/blackzhanzhan/Cyber-Ming-Protocol.git
2. 读取 `skill/global_rules/SKILL.md`、`skill/approval-first-planner/SKILL.md`、`skill/approved-checklist-executor/SKILL.md`
3. 按你当前宿主支持的项目级 skill 机制接入这三件套
4. 只回报：是否支持、如何接入、当前哪几份文件被当作实际法统

如果你的宿主不支持项目级 skill，请明确说“IDE 端稳定闭环当前不成立”，不要伪造安装成功。
```

### 第一步：先装核心三件套

- `global_rules`
- `approval-first-planner`
- `approved-checklist-executor`

这三者解决最核心的问题：先规划、再执行、按片归档。

### 第二步：按需要补扩展件

- `probe-first-scout`：面对真不确定，不是假复杂时
- `legacy-project-handover`：新窗接手、上下文续命或只读交接时

### 为什么这里不写"一键安装"

因为当前不同 IDE / agent host 对 skill 的装载方式并不统一。这个仓库提供的是可公开分发的 skill definitions，不是绑定单一宿主的安装脚本。

## 最小使用顺序

即使接入 Skill，IDE 端最稳妥的最小顺序仍然是：

1. 先让 planner 产出原子执行合同与边界
2. 让人类审核并决定是否准奏
3. 准奏后再让 executor 按片执行、验证、归档
4. 但请记住：这还必须配合 Web 端固定提示词，双端一起才叫最小稳定闭环

## 如果你的环境不支持 Skill

那就先不要被安装问题卡住。

你仍然可以完整实践协议：手工要求 planner 风格的输出、手工要求 executor 风格的执行与证据回报、手工用 Web 模板组织审计位提问。

**协议优先于安装。**

## 最常见的坑

### 坑一：一装 Skill 就想跳过理解

这样通常只会把误用变得更稳定。

### 坑二：一装 Skill 就想跳过审批

不对。Skill 强化的是动作骨架，不是用来绕开人类批准与证据门槛。

### 坑三：把 Web 模板也塞进本地 Skill 流程

不对。Web 模板属于 Web 审计位的协作资产，不属于 IDE 执行位的本地安装物。

### 坑四：把 Skill 当成交钥匙的一键外挂

Skill 只是强化器。真正决定是否成立的，仍然是：是否先审批、是否按片执行、是否有证据闭环、是否保留人类居中裁决。
