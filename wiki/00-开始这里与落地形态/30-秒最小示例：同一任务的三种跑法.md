# 30 秒最小示例：同一任务的三种跑法

## 这一页解决什么问题

前面的页面已经把协议、Skill、Web 审计模板之间的关系讲清了。

但很多人真正还差最后一步：

**我脑子里到底怎么快速跑一遍？**

这一页就只做这件事。

它不用大任务举例，只用一个足够小、足够普通、能在 30 秒里脑补完一轮的任务来演示三种跑法：

- 不装 Skill，手工跑
- 装了 Skill，更稳地跑
- Web 审计位怎样固定在循环里

## 示例任务

这次固定用一个很小的任务：

> 修一个设置页保存后不显示成功提示的小 bug。

为什么选这个？因为它足够小，不会把你脑中的最小示例拖成一场大重构；但它又足够真实，仍然能看到方案、执行、验收、复核这几步是怎么接上的。

## 三种跑法总览

| 跑法 | 你先做什么 | 它最能说明什么 | 不是在说明什么 |
| --- | --- | --- | --- |
| 不装 Skill | 先让 IDE 交原子清单 | 协议本身就能手工实践 | 没 Skill 就不能用 |
| 装了 Skill | 用 skill 稳定 planner / executor 节奏 | Skill 是强化器 | Skill 会自动产生完成事实 |
| Web 审计固定在循环里 | 把计划先交给 Web 审，再决定是否执行 | 审计流程不丢失，同时减少人类反复组织流程的精力损耗 | Web 位会变成第二个执行位 |

## 第一种：不装 Skill，手工跑

### 第一步：先让执行位交方案，不准直接开工

你可以直接对 IDE 说：

```text
我要修一个设置页保存后不显示成功提示的小 bug。

先别直接改代码。
先给我一个尽可能细的原子清单，告诉我准备改哪一层、怎么验收。
```

这一步的目标，不是让 IDE 显得谨慎，而是让它先暴露自己准备怎么做。

如果它能拆成类似下面这种粒度，就已经够你继续了：

- 定位保存成功条件为什么没有触发
- 修复成功提示出现的判断或状态更新
- 补一条最小回归检查，确认保存后确实出现成功提示

### 第二步：先把清单送 Web 审，再决定是否执行

当方案像样以后，不要立刻让它开改，而是先把这份清单原样复制给 Web，再补一句：

```text
这是执行位的方案。这个执行位可能骗我。
请你先看：有没有漏步、有没有把事情说得太顺、最后我要看什么证据。
```

如果 Web 放行，你再让执行位开始改；如果 Web 说这里有缺口，就把它的审核意见原样回给执行位，让它改方案后继续这个循环。

### 第三步：做完后，结果与证据再回 Web 验收

做完后，不要只收一句“已经修好”，而要收这几样：

- 当前运行的测试结果或回归检查结果
- 能证明成功提示真的出现的截图、日志或页面结果
- 对应这一步改动的提交记录

你在这里实践的，其实就是 Cyber-Ming 的最小骨架：

- 先审批
- 再执行
- 最后凭证据验收

到这一步为止，你还完全没有装 Skill，但你已经在使用协议。

## 第二种：装了 Skill，更稳但不替代证据

现在把同一个任务放到已经接入 Skill 的环境里看。

变化不在于任务 magically 完成，而在于高频动作更容易稳定出现。

### planner 更容易先交对的东西

像 `approval-first-planner` 这类 skill，更容易把任务先压成：

- 原子片
- 边界
- 红灯 / 绿灯
- 验收阶梯

这时你更少需要每次手工重复整段长提示。

### executor 更容易守住节奏

像 `approved-checklist-executor` 这类 skill，更容易把后续动作守成：

- 按批准过的片去做
- 做完先验证
- 验完再归档

这会减少“明明知道该这么做，但每次都说不全”的损耗。

### 但最重要的一条完全没变

无论 Skill 触发得多稳定，你最后仍然不能只看格式漂亮的汇报。

你还是要看：

- 当前运行的测试或检查输出
- 成功提示确实出现的证据
- 对应这次状态跃迁的提交记录

也就是说，Skill 让动作更稳，但不替代证据。

## 第三种：Web 审计位固定在循环里

这一条不是可选增强，而是这套协议在小任务上也不丢审计流程的关键。

### Web 位先看什么

先看执行位交上来的原子清单，并明确告诉它：这里可能有诈。这样 Web 位就不会顺着执行位的乐观叙事走，而会先查：

- 有没有漏步
- 有没有把难点说粗
- 最后到底要看什么证据

### Web 没通过时怎么走

如果 Web 说方案有缺口，不要跳过它，也不要自己脑补修正，而是把审核意见原样回给执行位，让它修改方案，然后继续下一轮方案审。

这一步真正减少的是人类精力损耗：你不用每次重新设计整套流程，只要持续扮演物理路由器，把方案和审核意见来回回灌。

### Web 通过后还要做什么

Web 放行以后，执行位才按清单执行、验证、提交。

执行完成后，你还要把下面这些材料再送回 Web：

- 本轮测试结果、截图、日志
- 对应提交记录
- 你自己最怀疑的风险点

如果是审方案，优先参考 `web-audit-templates/plan_audit_template.md`；如果是审完成，优先参考 `web-audit-templates/completion_audit_template.md`。

### Web 位不做什么

真正重要的是它不该做什么：

- 不替你继续写代码
- 不替执行位顺手扩需求
- 不靠总结陈词直接放行

它的职责是独立复核，不是第二个执行位。

## 看完这一页后，你应该立刻能回答的三句话

1. **不装 Skill，也能跑一轮 Cyber-Ming。**
2. **装了 Skill，会让动作更稳，但不会替代证据。**
3. **无论任务多小，原子清单都先送 Web 审；不过就回灌执行位改方案，通过后再执行提交，结果与证据还要再回 Web 验收。**

这页的职责到这里就结束了：它负责让你脑中先跑通一轮，不负责替代后面的路线页和法统正文。
