# 上手路线图：纯阅读、手工实践、Skill 增强、Web 审计

## 这一页解决什么问题

不是每个进入 Cyber-Ming 的人都应该走同一条路。

有人先需要判断这套协议值不值得学；有人已经准备动手；有人只想让 planner / executor 更稳定；也有人已经有执行位，只差一个独立审计位。

这一页的目标，就是帮你根据当前成熟度选路，而不是强迫你一步到位。

## 四条路线总览

| 路线 | 适合谁 | 第一目标 | 第一站 | 先不要急着做什么 |
| --- | --- | --- | --- | --- |
| 纯阅读 | 先判断是否认同这套治理逻辑的人 | 看清问题与位置 | `README.md`、`wiki/01-哲学与坐标/` | 不要急着装 Skill |
| 手工实践 | 想先用现有工具跑通一次闭环的人 | 跑通方案审、执行、验收 | `wiki/02-最小闭环与核心礼法/` | 不要急着追求自动化 |
| Skill 增强 | 已理解协议骨架、想减少漏步骤的人 | 稳定触发高频动作 | `skill/` 与本页的 Skill 路径 | 不要把 Skill 当裁决器 |
| Web 审计 | 已有执行位、需要独立审计位的人 | 建立方案审、完成审与续命判 | `web-audit-templates/` | 不要让 Web 位接管主线施工 |

如果你现在还拿不准自己属于哪一类，最短默认顺序是：先看一个 30 秒示例，再回到这页选路。

- 偏小 bug：[`30 秒最小示例：同一任务的三种跑法`](30-秒最小示例：同一任务的三种跑法.md)
- 偏小 UI / 文案：[`30 秒最小示例：小 UI 与文案改动怎么跑`](30-秒最小示例：小 UI 与文案改动怎么跑.md)

如果你现在不是想“读完再说”，而是准备直接把仓库交给执行位与审计位，就先看 [`让执行位与审计位自举入场`](让执行位与审计位自举入场.md)。

## 路线 A：纯阅读理解

### 适合什么状态

- 你还在判断这是不是一套真的能落地的方法
- 你对“主权、审计、续命、分封”这些词还很陌生
- 你更想先看理论与证据，而不是马上改工作流

### 建议顺序

1. 先读 `README.md`
2. 再读 `wiki/01-哲学与坐标/`
3. 最后看 `wiki/04-战报与样本/`

### 到什么程度算读够了

当你已经能回答下面三个问题，就可以进入下一条路：

- 为什么深水区里“能生成”不等于“能治理”
- 为什么执行位和审计位最好分开
- 为什么完成事实必须靠证据而不是总结

## 路线 B：无 Skill 手工实践

### 适合什么状态

- 你已经基本认同这套问题定义
- 你想先拿现有工具跑通一遍，不急着改装环境
- 你更关心协议是不是实用，而不是 Skill 好不好装
- 或者你已经认真用过 workflow / spec-driven 开发，但开始感觉项目形态被过早钉死

### 建议顺序

1. 读 `wiki/02-最小闭环与核心礼法/最小闭环：一次审计版与多次审计版.md`
2. 读 `wiki/02-最小闭环与核心礼法/核心礼法之一：原子级任务清单与赛博起居注.md`
3. 读 `wiki/02-最小闭环与核心礼法/白盒物理对账：什么算完成事实.md`
4. 用你现有的 IDE 与 Web 工具手工跑一次最小闭环

如果你在第 1 步前还觉得“流程感”不够具体，就先插读那篇 30 秒示例，再回来正式走这一条路。

如果你最强的痛点不是黑盒失控，而是 workflow 太早把项目冻住，优先去看 [`核心礼法之一：原子级任务清单与赛博起居注`](../02-最小闭环与核心礼法/核心礼法之一：原子级任务清单与赛博起居注.md)。

### 第一里程碑

你能亲手完成下面这件事：

- 让执行位先交原子清单
- 让 Web 位先审方案再放行
- 施工结束后再用证据送审

这时你已经不是“只懂理念”，而是已经在实践协议。

## 路线 C：Skill 增强执行

### 适合什么状态

- 你已经知道审批先行、原子清单、证据闭环为何存在
- 你发现自己总在高频任务里漏说边界、漏收证据、漏切片
- 你想让执行动作更稳定，而不是更黑盒

### 建议顺序

1. 先看 [`什么时候装 Skill，什么时候先别装`](什么时候装-Skill，什么时候先别装.md)
2. 再看 [`安装与最小使用`](安装与最小使用.md)
3. 先只装最核心的一组 skill，再跑一轮最小闭环

如果你想先感受“没装 / 装了”到底差在哪，而不想先吞完所有说明，也可以先插读那篇 30 秒示例。

### 第一里程碑

你能稳定触发这些动作，而不需要每次手工重复长提示：

- 先出规划，再等批准
- 批准后按片执行
- 先探测，再扩刀
- 需要时产出接手快照

## 路线 D：Web 审计协作

### 适合什么状态

- 你已经有一个 IDE 执行位
- 你缺的是一个独立、窄上下文、非执行导向的审计位
- 你想把方案审、完成审、续命判断更稳定地外包给 Web 侧

### 建议顺序

1. 读 [`Web 审计模板怎么协作：它不是本地 Skill`](Web-审计模板怎么协作：它不是本地-Skill.md)
2. 对照 `web-audit-templates/plan_audit_template.md`
3. 对照 `web-audit-templates/completion_audit_template.md`
4. 对照 `web-audit-templates/succession_judge_template.md`

### 第一里程碑

你能把 Web 位稳定放在下面的位置上：

- 审计而不是施工
- 查证据而不是听捷报
- 判断是否该续命，而不是在旧上下文里跟着漂

## 如果你还拿不准，默认怎么选

最稳妥的默认路线是：

1. 先走 **纯阅读**
2. 再走 **无 Skill 手工实践**
3. 确认会长期采用后，再进入 **Skill 增强**
4. 需要独立审计位时，再把 **Web 审计** 补上

这条顺序的好处是：

- 不会把工具安装误当成协议理解
- 不会把 Skill 误当成协议本体
- 不会把 Web 模板误当成本地安装物
