## 这一页解决什么问题

很多人看到 `web-audit-templates/` 后会本能地问：

- 这些是不是也该装进本地 skill 目录
- 它们和 `skill/` 里的东西到底有什么不同
- Web 位是不是也应该顺手接管施工

这页就是把这些边界一次讲清。

## 先讲结论

`web-audit-templates/` 里的文件是 **Web-side 审计协作模板**。

它们的作用是帮助你在 Web 侧更稳定地：

- 审方案
- 审完成
- 判续命

它们不是本地 Skill，不是 IDE 执行位的安装物，也不是拿来直接施工的东西。

## 三个模板分别做什么

### `plan_audit_template.md`

用于方案审计。

重点不是帮执行位写实现，而是判断：

- 原子片是不是够小
- 红绿灯是不是够实
- 验收阶梯是不是够清
- 该不该先缩刀或先做探测

### `completion_audit_template.md`

用于完成审计。

重点是查：

- 计划中的红绿灯是否真的执行过
- 所谓“通过”是不是解决了同一个问题
- 证据是否对应当前运行与当前提交
- 这是完成事实，还是伪完成

### `succession_judge_template.md`

用于续命判断。

重点是查：

- 当前窗口是否已经腐烂
- 这是不是惰性修补与打地鼠漂移
- 应该继续、打断，还是异步续命

## 它为什么不是本地 Skill

因为它服务的不是 IDE-side 执行位，而是 Web-side 审计位。

协议要求的是角色分离：

- 执行位负责施工、验证、归档
- 审计位负责盘问、查证、裁断
- 人类居中负责路由与最终裁决

如果把 Web 模板也理解成“本地技能包”，最容易导致两种退化：

- 在同一执行上下文里自己审自己
- 让 Web 位顺手接管施工，丢失独立审计视角

## 正确的协作方式

### 第一步：人类打包最小材料

不要把整个上下文原样倾倒给 Web 位，而是有选择地提供：

- 当前任务摘要
- 原子清单或当前片定义
- 关键日志、工件、截图、提交记录
- 你怀疑的风险点

### 第二步：选对模板

- 审方案，用 `plan_audit_template.md`
- 审完成，用 `completion_audit_template.md`
- 判续命，用 `succession_judge_template.md`

### 第三步：要求 Web 位只做审计，不做施工

真正重要的是角色位置，而不是角色名称。

Web 位最有价值的地方，在于它：

- 不背 IDE 执行位那条长上下文
- 不直接沉浸在写码惯性里
- 更适合站在证据与边界角度查错

## 最容易出现的误用

### 误用一：把 Web 模板和 Skill 一起安装

不对。
Web 模板要做的是协作，不是安装。

### 误用二：把 Web 位变成第二个执行位

不对。
如果 Web 位开始直接设计实现、替执行位补代码、顺手接管主线，它的独立审计价值就会下降。

### 误用三：不给 Web 位最小证据包，只让它听总结

不对。
Web 位不是来听捷报的，而是来查证据的。
