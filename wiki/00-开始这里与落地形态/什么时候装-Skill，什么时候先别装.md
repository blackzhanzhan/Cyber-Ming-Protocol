# 什么时候装 Skill，什么时候先别装

## 这一页解决什么问题

Skill 加进项目后，最容易出现的一种误判就是：

> 既然仓库里已经有 skill，那我是不是应该立刻全装？

不一定。

Cyber-Ming 的正确顺序不是“先装，再理解”，而通常是“先理解，再决定装到什么程度”。

## 先给结论

### 适合先不装 Skill 的情况

- 你还在理解协议，尚未确认会长期采用
- 你只是想先读理念、战报与边界
- 你想先用自己现有的 IDE / Web 组合手工跑一遍
- 当前任务很小、很轻、很短，不值得引入完整治理骨架

### 适合开始装 Skill 的情况

- 你已经理解审批先行、原子清单、证据闭环三件事
- 你发现自己经常漏步骤、漏边界、漏证据
- 你想稳定 planner / executor / probe / succession 的动作
- 你经常在 AI coding 里遇到伪完成、惰性修补、上下文腐烂

### 暂时不建议全装的情况

- 你的宿主环境对 skill 触发支持还不稳定
- 你还没想好 IDE 执行位与 Web 审计位怎么分工
- 你目前只需要最小闭环，不需要完整的技能面板

## 为什么不建议一上来就全装

因为 Skill 解决的是**高频动作稳定性**，不是**协议理解缺失**。

如果你连下面这些问题都还没想清：

- 为什么要先审方案再开工
- 为什么执行位不应自证完成
- 为什么完成必须靠证据闭环
- 为什么窗口腐烂后要续命

那么你装再多 Skill，也很容易只是在更稳定地误用它们。

## 最稳妥的接入顺序

### 第一阶段：先手工跑通

目标不是快，而是知道协议到底在防什么。

这时最重要的是：

- 你能亲手要求原子清单
- 你能亲手组织 Web 审计
- 你能亲手根据证据裁定是否完成

### 第二阶段：只装核心 Skill

当你已经知道自己会长期用，就先装最核心的一组：

- `global_rules`
- `approval-first-planner`
- `approved-checklist-executor`

这三者先把“先规划、后执行、按片归档”这条主干立住。

### 第三阶段：按需要补装扩展 Skill

当你的任务更复杂、更不确定、更长程时，再补：

- `probe-first-scout`
- `legacy-project-handover`

前者解决“先探测，不装懂”；后者解决“新窗接手、上下文续命、只读快照”。

## 一个简单判断法

如果你最近的主要痛点是：

- **不确定值不值得学**：先别装
- **知道该怎么做，但总忘记说全**：可以开始装
- **环境还很乱，连角色分工都没定**：先别全装
- **已经在复杂项目里反复踩伪完成和腐烂上下文**：尽快装核心 Skill

## 最常见的两种误装方式

### 误装一：把 Skill 当成交钥匙的一键外挂

这会把项目误读成“只要装了，协议就自动生效”。

实际上，Skill 只是强化器。真正决定是否成立的，仍然是：

- 是否先审批
- 是否按片执行
- 是否有证据闭环
- 是否保留人类居中裁决

### 误装二：把 Web 模板也当成要一起安装的技能包

这会把角色边界搞混。

`web-audit-templates/` 不是本地 Skill，它们服务的是 Web 审计位，不是 IDE 执行位的安装流程。
