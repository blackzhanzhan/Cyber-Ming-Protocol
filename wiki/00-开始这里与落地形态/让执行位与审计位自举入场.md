## 这一页解决什么问题

如果你已经理解这套协议，下一步最省力的尝试方式，往往不是自己把全套规则重新解释给两个 agent，而是直接把仓库与角色文件交给它们，让它们循仓自举入场。

但这条浅尝试入口更适合已经读过核心思想、对基本流程有直觉认知的人。若你还没有这层认知，先回 `wiki/Home.md` 或 `00` 前置层，再来授位会更稳。

还要再记一条限制：由于模型能力有差异，尤其在没有安装项目 skill 的情况下，这个一键授位不能保证 agent 立刻严格守住角色边界。已经观察到有模型会先承认“开工前先交清单”，但真正收到需求后仍直接写代码。

所以你不一定要完整采用手工做法，但你必须理解手工做法的流程。因为这正是你判断何时该打断、为什么该打断、何时行使中断权与圣裁权的依据。

这页回答的就是：

- 仓库链接该分别交给谁
- 执行位和审计位各自先读什么
- 它们第一轮应该怎么回
- 什么情况才算真正进入协议

## 最短操作法

1. 把当前仓库链接同时交给 IDE 执行位和 Web 审计位
2. 给执行位：[`../../BOOTSTRAP.md`](../../BOOTSTRAP.md) + [`../../bootstrap/ide-executor.md`](../../bootstrap/ide-executor.md)
3. 给审计位：[`../../BOOTSTRAP.md`](../../BOOTSTRAP.md) + [`../../bootstrap/web-auditor.md`](../../bootstrap/web-auditor.md)
4. 浅尝试默认不 clone，先让执行位把仓库当远程法统来源来读；若宿主支持 URL 读取、`webfetch` 或浏览器仓库阅读，优先这样做
5. 第一轮先不要发案卷，先只看它们各自的入场确认
6. 第二轮再发送本轮案卷材料
7. 第三轮才开始正式审案与执行循环

## 他们第一轮应该怎么回

### 执行位第一轮

至少要明确四件事：

- 它知道自己是执行位
- 它不会先改文件
- 它会先交原子清单与边界
- 它知道方案必须先送 Web 审，再进入执行

### 审计位第一轮

至少要明确四件事：

- 它知道自己是审计位
- 它只审方案与证据，不代替执行
- 它会检查伪完成、漏步、假证据与偷换目标
- 它会先说明自己需要的输入包是什么
- 它不会在这一轮直接评论当前会话里的具体补丁、编号、测试结论或旧案卷

## 什么算入场成功

只有当下面这些条件都成立，才算真正自举成功：

- 执行位能说清谁执行、谁审计、谁裁决
- 审计位也能说清同一套角色分工
- 执行位的第一步是交清单，不是直接施工
- 审计位的第一步是审计划或证据，不是顺手接管实现
- 双方都知道：完成不能靠执行位自报，而要靠证据与 Web 验收成立

打断不是失败，而是这套协议里的正常权力动作。你越理解手工流程，越知道什么时候该让执行位停下、退回、重交方案，而不是等它一路滑到伪完成再来收拾。

## 怎么识别伪自举

如果 Web 审计位在第一轮里就直接出现下面这些东西，就不要把它当成已经成功入场：

- 直接引用当前会话里的补丁方案、测试编号、模块名、旧结论
- 直接表现出它“认识你是谁”或记得你别的对话、旧项目、历史案卷
- 直接批红或放行某个具体方案
- 直接开始评论“这一步该怎么修”而不是先确认角色与路由

这种情况不叫“已经理解得很快”，而更可能是旧会话残影或平台记忆先接管了输出。你应把它视为**伪自举**，重新清场后再授位一次。

如果只靠 prompt 清场效果还是不好，就先去宿主设置里暂时关闭用户记忆 / 个性化 / 历史对话学习之类的选项，再新开对话重试。
