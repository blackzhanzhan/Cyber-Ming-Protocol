# 让执行位与审计位自举入场

## 这一页解决什么问题

如果你已经理解这套协议，下一步最省力的尝试方式，往往不是自己把全套规则重新解释给两个 agent，而是直接把仓库与角色文件交给它们，让它们循仓自举入场。

这页回答的就是：

- 仓库链接该分别交给谁
- 执行位和审计位各自先读什么
- 它们第一轮应该怎么回
- 什么情况才算真正进入协议

## 最短操作法

1. 把当前仓库链接同时交给 IDE 执行位和 Web 审计位
2. 给执行位：[`../../BOOTSTRAP.md`](../../BOOTSTRAP.md) + [`../../bootstrap/ide-executor.md`](../../bootstrap/ide-executor.md)
3. 给审计位：[`../../BOOTSTRAP.md`](../../BOOTSTRAP.md) + [`../../bootstrap/web-auditor.md`](../../bootstrap/web-auditor.md)
4. 第一轮先不要发案卷，先只看它们各自的入场确认
5. 第二轮再发送本轮案卷材料
6. 第三轮才开始正式审案与执行循环

## 给执行位什么

- 当前仓库链接
- [`../../BOOTSTRAP.md`](../../BOOTSTRAP.md)
- [`../../bootstrap/ide-executor.md`](../../bootstrap/ide-executor.md)

如果你只是浅尝试，这三样就够了。

## 给审计位什么

- 当前仓库链接
- [`../../BOOTSTRAP.md`](../../BOOTSTRAP.md)
- [`../../bootstrap/web-auditor.md`](../../bootstrap/web-auditor.md)

如果你希望它更快进入审计状态，也可以补给它 `web-audit-templates/` 中对应模板，但角色文件本身已经足以完成第一轮入场。

## 他们第一轮应该怎么回

### 执行位第一轮

至少要明确四件事：

- 它知道自己是执行位
- 它不会先改文件
- 它会先交原子清单与边界
- 它知道方案必须先送 Web 审，再进入执行

### 审计位第一轮

至少要明确四件事：

- 它知道自己是审计位
- 它只审方案与证据，不代替执行
- 它会检查伪完成、漏步、假证据与偷换目标
- 它会先说明自己需要的输入包是什么
- 它不会在这一轮直接评论当前会话里的具体补丁、编号、测试结论或旧案卷

## 什么算入场成功

只有当下面这些条件都成立，才算真正自举成功：

- 执行位能说清谁执行、谁审计、谁裁决
- 审计位也能说清同一套角色分工
- 执行位的第一步是交清单，不是直接施工
- 审计位的第一步是审计划或证据，不是顺手接管实现
- 双方都知道：完成不能靠执行位自报，而要靠证据与 Web 验收成立

## 怎么识别伪自举

如果 Web 审计位在第一轮里就直接出现下面这些东西，就不要把它当成已经成功入场：

- 直接引用当前会话里的补丁方案、测试编号、模块名、旧结论
- 直接批红或放行某个具体方案
- 直接开始评论“这一步该怎么修”而不是先确认角色与路由

这种情况不叫“已经理解得很快”，而更可能是旧会话残影先接管了输出。你应把它视为**伪自举**，重新清场后再授位一次。
